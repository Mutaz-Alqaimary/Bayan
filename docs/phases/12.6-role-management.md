## Goal

Let an **admin** manage **teacher** accounts entirely from inside the application — promote an
existing authenticated user (a student) to teacher, and demote a teacher back to student — by
flipping **`profiles.role`** only. Admin accounts stay infrastructure-only (never created or assigned
in-app). No schema change, no new `teachers` table, no change to the Phase 12.5 identity model.
Alongside this, **Part 1** lets any signed-in user edit only their own `full_name` and avatar, with a
redesigned RLS policy that makes every other `profiles` column (including `role`) unwritable by
clients.

## Part 1 — Profile Editing (added to Phase 12.6, implemented first)

### P1.1 Why Profile Editing was introduced

The original 12.6 security patch proposed to *drop* the over-broad `profiles_update_own` policy
(no app code self-updated `profiles`). The owner instead chose to add a **minimal, legitimate
self-edit** (display name + avatar) and **redesign** the policy to be safe — so users get a basic
profile while `role` and every identity/authorization field stay locked. This makes the
`profiles_update_own` redesign the right long-term shape rather than a temporary removal.

### P1.2 Scope (locked)

- **Editable by the user (only):** `full_name`, `avatar_url`.
- **Never editable by the user:** `role`, `id`, `email` (admin/teacher-managed via Student
  Management), `student_number`, `grade` (these live on `students`, not exposed here), `locale` /
  `theme` / `reduced_motion` / `email_notifications` (already managed by Settings on `user_settings`),
  and **any** future authorization/identity field.
- **No new page.** A "Profile" card is added at the top of the existing `features/settings`
  `SettingsPage`, as a **separate form** from the four-preference settings form (different table,
  different save path). Email and Role are shown **read-only** for context.

### P1.3 `avatar_url` stores the object **path**, not a URL (owner decision #1)

`profiles.avatar_url` stores the **storage object path** exactly as
`avatars/{user_id}/avatar.webp` — **never** a full public/CDN URL. The application generates the
display URL **at runtime** (`supabase.storage.from('avatars').getPublicUrl(objectPath)`, with a
cache-bust derived from `profiles.updated_at`), and can switch to signed URLs or a new CDN/domain
later **without touching stored data**. A helper splits the stored value into bucket (`avatars`) +
in-bucket key (`{user_id}/avatar.webp`).

### P1.4 Avatar pipeline (owner decisions #2, #3)

- **WebP passthrough:** if the uploaded file is already `image/webp`, **upload it directly**. Only
  **JPEG/PNG** are converted to WebP (client-side via `<canvas>.toBlob('image/webp')` — no new
  dependency, respecting the locked stack). Other types are rejected.
- **Max size 1 MB** (owner decision #3) — enforced client-side *and* by the bucket `file_size_limit`
  (defense in depth).
- **Stable path = zero orphans:** the object is always `avatars/{user_id}/avatar.webp`, so replace is
  an `upsert` overwrite — there is never a previous file to orphan. **Remove** nulls `avatar_url`
  then best-effort deletes the object.
- Required UX: client validation (type + size), **preview**, **loading** state, success toast,
  localized **error** handling.

### P1.5 Upload sequence — profile data is NEVER updated before upload succeeds (owner decision #4)

Strict ordering; each step only runs if the previous succeeded:

1. **Select** a file (`AvatarUploader`, client).
2. **Validate** MIME (`image/webp|jpeg|png`) and size (≤ 1 MB). Fail → localized error, **nothing
   uploaded, profile untouched.**
3. **Normalize to WebP:** already-WebP → use as-is; JPEG/PNG → canvas-convert to a WebP `Blob`.
   Conversion failure → error, **abort, profile untouched.**
4. **Preview** the local blob (object URL only — not persisted).
5. **Upload** the WebP to `avatars/{auth.uid()}/avatar.webp` (`upsert: true`, `contentType:
   'image/webp'`) via the browser client (Storage RLS confines it to the user's own folder).
   **Upload failure → error; `profiles` is NOT written** (avatar_url unchanged).
6. **Only after upload success →** call `updateProfileAction` to persist
   `avatar_url = 'avatars/{user_id}/avatar.webp'` (the **path**) via the **session** client (RLS +
   column privileges enforce). The action **derives the path itself** from `auth.uid()` and never
   trusts a client-supplied value.
7. **Display:** any avatar render reads the stored path and generates the URL at runtime
   (`getPublicUrl` + `?v={updated_at}` cache-bust).

> **Failure semantics:** because the path is deterministic and stable, a failure *after* a successful
> upload but *before* the profile write is self-healing — a retry overwrites the same object and
> re-persists the same path; there are no orphaned files and no divergence. Name edits and avatar
> edits are independent saves, so a failed avatar step never affects `full_name` and vice-versa.

### P1.6 Profiles RLS redesign — exact SQL (replaces the §3.0 "drop" decision)

RLS can't restrict columns, so column scope comes from **column privileges** and row scope from the
**policy** — two layers, each closing a distinct hole:

```sql
-- Layer 1 — COLUMN privileges (hard guarantee: role/id/etc. unwritable by clients)
revoke update on public.profiles from anon, authenticated;
grant  update (full_name, avatar_url) on public.profiles to authenticated;

-- Layer 2 — RLS row scope (authenticated, own row, redundant ownership check)
drop policy if exists "profiles_update_own" on public.profiles;
create policy "profiles_update_own"
on public.profiles
for update
to authenticated
using      ( auth.uid() = id )
with check ( auth.uid() = id );
```

**Why safest:** column privileges make `role`/`id`/`locale`/etc. **physically unwritable** by
`authenticated`/`anon` regardless of any present or future policy; the policy confines edits to the
caller's **own** row. No fragile `role = (select …)` predicate is used (redundant once `role` is
privilege-revoked). `service_role` is untouched, so role changes + admin email changes stay
server-only. Any **future** privileged column is unwritable by default (must be explicitly granted).

### P1.7 Avatar Storage — manual config + exact policies

**Manual (dashboard / Storage API — owner applies):** bucket **`avatars`** exists + **Public** read +
**`file_size_limit` = 1 MB** + **`allowed_mime_types` = `image/webp,image/png,image/jpeg`**.

```sql
-- storage.objects — owner-scoped writes, public read
create policy "avatars_owner_insert" on storage.objects for insert to authenticated
  with check ( bucket_id = 'avatars' and (storage.foldername(name))[1] = auth.uid()::text );
create policy "avatars_owner_update" on storage.objects for update to authenticated
  using      ( bucket_id = 'avatars' and (storage.foldername(name))[1] = auth.uid()::text )
  with check ( bucket_id = 'avatars' and (storage.foldername(name))[1] = auth.uid()::text );
create policy "avatars_owner_delete" on storage.objects for delete to authenticated
  using      ( bucket_id = 'avatars' and (storage.foldername(name))[1] = auth.uid()::text );
-- REQUIRED: the upsert upload path internally reads storage.objects (verified in production).
create policy "avatars_public_read" on storage.objects for select to public
  using ( bucket_id = 'avatars' );
```

> **Verified in production (corrects an earlier wrong assumption):** all **four** policies are
> **required** — including `avatars_public_read`. Removing the SELECT policy **breaks avatar
> upload/replace** (re-adding it restores them): an `upsert: true` upload makes Supabase Storage
> **internally read `storage.objects`** (existence check on the stable `avatar.webp` path), which RLS
> gates even though the app issues no read API call. The bucket must also remain **Public** for object
> display. See [`docs/database/manual-supabase-configuration.md`](../database/manual-supabase-configuration.md)
> §6a for the verified behavior and invariant.

### P1.8 New names (add to `naming-conventions.md`)

```
UpdateProfileFormValues / UpdateProfileResult / UpdateProfileMessages   // features/profile (or settings)
buildUpdateProfileSchema / useProfileSchemaMessages
updateProfileAction / removeAvatarAction                                // session-client, own row
avatarObjectPath(userId) / avatarPublicUrl(path, version)               // lib/storage.ts (avatar helper)
```

### P1.9 Why these constraints (documented rationale)

- **Why only Full Name + Avatar are editable:** they are the only non-authorization, non-identity,
  user-owned presentation fields. Everything else is either an authorization field (`role`), an
  identity field (`id`, `email`, `student_number`), an academic field (`grade`, on `students`), or
  already owned by Settings (`locale`/`theme`/`reduced_motion`/`email_notifications`).
- **Why `profiles_update_own` was redesigned, not removed:** the user now legitimately self-edits two
  fields, so a scoped, column-privilege-backed policy is the correct durable shape — safer than the
  original broad policy and more useful than dropping it.
- **Why role stays server-side only:** `role` drives authorization; it is privilege-revoked for
  clients and writable **only** by the role-gated server action via the service-role client.
- **Why Email stays admin-managed:** email is an identity + login attribute tied to `auth.users`;
  changing it is a privileged dual-write (Phase 12.5), never a self-service field.

## 1. Architecture

### 1.1 The core model — role lives on `profiles`, nothing else

The codebase already derives all authority from `profiles.role`
(`getSessionUser()` → `requireRole()` → `features/auth/roles.ts`). A "teacher" is therefore **exactly**
`profiles.role = 'teacher'` — the single source of truth. No `teachers` table is created (see §1.4).

Two distinct concepts already exist and must not be conflated:

- **A person** = `auth.users` + `profiles`. `profiles.role` makes them admin / teacher / student.
- **A student roster record** = a `students` row, which *may or may not* be linked
  (`profile_id`) to a person.

**Role management operates on the person (`profiles`), never on the roster (`students`).** Promoting a
student to teacher sets `profiles.role` from `'student'` to `'teacher'`; the linked `students` row, its
`profile_id`, reading history, and `user_settings` are **untouched**.

### 1.2 Only authenticated users can be promoted

A roster-only student (`profile_id IS NULL`, no `auth.users`/`profiles`) has **no role to change** and
**cannot** be promoted. The promotion candidate set is therefore **`profiles` where `role = 'student'`**
— *not* the `students` roster. The promote picker lists people (profiles), resolved with the
service-role client (cross-`profiles` reads cross RLS — see §5).

### 1.3 Allowed transitions (admin-only, student ↔ teacher only)

```
student ⇄ teacher        ✅ (admin only)
* → admin                ❌ never (infrastructure-only)
admin → *                ❌ never (admin profiles are not manageable in-app)
self (actor's own role)  ❌ never (no self role-change)
```

### 1.4 Why no `teachers` table

- Every consumer of "is this a teacher?" already reads `profiles.role`
  (e.g. `features/dashboard/data/admin.ts` counts teachers via `profiles … eq('role','teacher')`).
- A teacher has **no required attributes** today: name = `profiles.full_name`, email =
  `auth.users.email`, status = derived from `auth.users.last_sign_in_at`. There is no teacher-specific
  `NOT NULL` data to store.
- A second table would duplicate identity that already lives in `profiles` and create a competing
  source of truth for "who is a teacher" that can drift from `profiles.role`.
- It would require a schema change, which the project forbids.

**Future teacher metadata** (department, specialization, qualifications) — if ever needed — becomes a
separate **extension table keyed by `profiles.id`** (e.g. `teacher_profiles(profile_id PK/FK, …)`),
added in its own future phase. It does **not** change today's identity model. See §10.

### 1.5 New feature module

A dedicated, independent **`features/teachers/`** module (sibling of `features/students/`), kept
**separate from Student Management** (no generic "Users" module). Mirrors the proven students module
shape: `types.ts`, `queries.ts`, `actions.ts`, `schemas.ts`, `components/`.

## 3. Security

### 3.0 ⚠️ Pre-existing DB-level finding — `profiles_update_own` permits self-role-change

While grounding this phase, the canonical `supabase/setup.sql` was found to define:

```sql
create policy "profiles_update_own" on public.profiles
for update using ( auth.uid() = id );   -- no WITH CHECK, no column restriction
```

With Supabase's default column grants to the `authenticated` role and **no `WITH CHECK`** on this
policy, an authenticated user can update **their own** `profiles` row directly via the browser client
(`supabaseClient.from('profiles').update({ role: 'admin' }).eq('id', <ownId>)`) — i.e. **self-escalate
their role, including to `admin`** — completely bypassing the application's server actions and UI gates.

This is **independent of** Phase 12.6's design (which correctly forbids self/admin changes at the app
layer), but it **undermines requirement #1 at the database layer**. The live database is the source of
truth (per `CLAUDE.md`), so it must be hardened — a manual Supabase change requiring the owner's
approval (the assistant never applies it automatically).

> **✅ Resolution (owner-approved).** Rather than drop the policy, **Part 1 — Profile Editing**
> introduces a minimal legitimate self-edit (`full_name` + avatar) and **redesigns**
> `profiles_update_own` with **column privileges + a scoped policy** so `role` (and every other
> column) becomes unwritable by clients. See **§P1.6** for the exact, approved SQL. That redesign —
> not "Option A/B" above — is the path being taken. Broad RLS tightening of the other permissive
> tables remains **Phase 17** scope; this specific admin-invariant gap is closed now.

### Application-layer controls

| Concern | Control |
|---|---|
| Escalation to admin | UI offers only `student\|teacher`; action rejects `admin` as source **or** destination; no create/promote-admin path anywhere. |
| Untrusted input | Zod-validate `{ profileId: uuid, newRole: enum('student','teacher') }`; re-validate server-side (Server Actions are POST-reachable). |
| Acting without authority | Every action begins `requireRole('admin')`; the `canManageTeachers` UI gate is defense-in-depth only. |
| RLS-bypass misuse | Service-role client used **only** inside the admin-gated action for the single `profiles` UPDATE; `import "server-only"`; never client-imported. |
| Self-lockout / confusion | Action forbids the actor changing **their own** role. |
| Stale UI / races | UPDATE guarded with `… AND role = <expectedCurrentRole>`; zero rows → safe localized error, no change. |
| Identity drift | Action writes **only** `profiles.role` (+ `updated_at`); never `profile_id`, `students`, `auth.users`, `user_settings`, `reading_sessions`, or sessions. |
| Surprising new access | Promotion grants only what `role='teacher'` already implies via existing gates; `profiles` RLS stays select-own. |
| Auditability | Server-side log of every role change. A first-class audit table is **future scope** (§10), noted as deferred — not silently skipped. |

## 5. Database impact

- **Schema:** zero changes. No tables, columns, constraints, indexes, triggers, or migrations.
  `profiles.role` already accepts `'teacher'`.
- **Writes introduced:** one statement only —
  `UPDATE public.profiles SET role = <'teacher'|'student'>, updated_at = now() WHERE id = <profile_id> AND role = <expected>`.
  No inserts/deletes; no writes to `students`/`auth.users`/`user_settings`/`reading_sessions`.
- **Access path:** `profiles` RLS is **select-own** with **no UPDATE policy**, so updating *another*
  user's role must go through the **service-role admin client** (bypasses RLS), gated by
  `requireRole('admin')`. This is the same role-gated service-role pattern the admin dashboard already
  uses for cross-`profiles` reads.
- **⚠️ Pre-implementation verification (read-only — see §6 / §7):** confirm `service_role` holds
  **UPDATE on `public.profiles`** (only `students` privileges were previously verified).

## 6. Migration strategy

There is **no data migration** — no schema change and no backfill. "Migration" here is the rollout
sequence (§2). The only environmental precondition is the **read-only privilege verification** in §7,
which must pass (or be remediated by you manually) **before** any code is built.

Rollback is trivial and lossless at every layer: each role change is a single reversible
`profiles.role` UPDATE; the new module is additive (new route/nav/feature folder) and removable
without touching identity or the schema.

## 7. Required read-only verification (gating precondition)

Run this in the **Supabase SQL editor** (read-only; changes nothing):

```sql
select privilege_type
from information_schema.role_table_grants
where grantee = 'service_role'
  and table_schema = 'public'
  and table_name = 'profiles'
order by privilege_type;
```

- **`UPDATE` present** (expected default) → no manual Supabase configuration required; proceed to
  implementation.
- **`UPDATE` absent** → this is the **only** manual Supabase action 12.6 would need. Per project rules
  the assistant **STOPS and explains**; it never runs a grant automatically. The remediation (to be
  approved and applied by the project owner) would be:
  ```sql
  -- Only if the check above shows UPDATE missing. Apply manually after approval.
  grant update on public.profiles to service_role;
  ```
  Record the outcome in `docs/database/manual-supabase-configuration.md` either way.

**Second check — the `profiles_update_own` self-escalation finding (§3.0).** Confirm the live policy
and decide whether to harden now or defer to Phase 17:

```sql
select polname, polcmd, pg_get_expr(polqual, polrelid)  as using_expr,
       pg_get_expr(polwithcheck, polrelid)               as check_expr
from pg_policy
where polrelid = 'public.profiles'::regclass;
```

If `profiles_update_own` exists for `UPDATE` with a `using` of `auth.uid() = id` and a **null
`check_expr`**, the self-role-change escalation is live and should be hardened (manual change,
owner-approved — §3.0 Options A/B).

> The assistant cannot reach `information_schema`/`pg_policy` from application tooling (PostgREST/
> `supabase-js` expose only the `public` schema, and there is no direct Postgres/`psql` connection
> string in the documented env). Both checks are therefore performed by the project owner in the
> Supabase SQL editor, and implementation is gated on their results.

## 8. Manual testing strategy

**Promotion**
- Admin promotes a `role='student'` user → appears in Teacher Management; `profiles.role='teacher'`;
  the user's `students` row, `profile_id`, `user_settings`, and reading history are unchanged.
- Promoted user, on next load, has teacher access (students/content/analytics nav) and loses the
  student-only reading workflow (`requireRole('student')` blocks it).

**Demotion**
- Admin demotes a teacher → back to `role='student'`; their previously-linked `students` row + reading
  history re-attach (lossless); identity unchanged.

**Admin invariant**
- No UI anywhere offers "admin" as a target. Calling the server action with `newRole='admin'`, or
  targeting an admin profile, is rejected. Acting on **self** is rejected.

**Eligibility**
- Roster-only students (no profile) never appear in the promote picker.

**Concurrency / stale UI**
- Two admins demoting the same teacher: the second updates zero rows and shows a safe error; no
  duplication, no error leak.

**Dual presence (UI)**
- A promoted teacher still owning a `students` row is clearly flagged in Student Management
  (badge/filter); admins are not confused; no data moved.

**States / i18n / a11y**
- Teacher list: loading skeleton, empty (CTA to promote), error (retry). RTL + dark mode correct;
  full keyboard nav; screen-reader labels; Arabic-aware search/sort.

**Authorization**
- Teacher and student cannot see the Teachers nav item, cannot open `/teachers` (route guard
  redirects), and cannot invoke the actions (server `requireRole('admin')`).

## 9. Risks & trade-offs

| Risk / trade-off | Assessment & mitigation |
|---|---|
| **Dual presence** (promoted teacher still a `students` row with history) | *Intentional* and required for lossless demotion. Mitigate with a display-only badge/filter (§2.9). Never "fix" by deleting the roster row — that destroys history and violates the constraint. |
| **`service_role` may lack UPDATE on `profiles`** | The only thing that could need a manual grant. De-risked by the §7 read-only check **before** building; assistant stops and flags rather than running a grant. |
| **Listing all auth users** for teacher email/status each load | Same pattern/scale as the existing student-status derivation; fine at school scale; revisit with caching/pagination in **Phase 14** (consistent with the existing deferral). |
| **No audit trail** for role changes | Acceptable now (server-side logging); a real audit log is future scope — documented, not skipped. |
| **Over-engineering pressure** (teacher fields, generic Users area) | Explicitly out of scope; 12.6 stays minimal (role flip + listing + separate module). |
| **Scope creep into 12.5 identity** | Hard rule: 12.6 writes **only** `profiles.role`; never identity columns. |

## Future Considerations

This section exists so the architectural intent survives beyond this phase. Future Claude Code
sessions must read it before extending role/teacher functionality.

### Architectural decisions made today (and why)

1. **Role is a single column on `profiles` — no `teachers` table.** Chosen because identity already
   lives in `profiles`, a teacher needs no extra required data today, and a second table would create
   a competing, drift-prone source of truth and force a forbidden schema change. *Simplicity and a
   single source of truth.*
2. **Role management mutates only `profiles.role`.** Chosen to keep the Phase 12.5 identity invariant
   (`auth.users.id ↔ profiles.id ↔ students.profile_id`) completely untouched and to make every change
   reversible and lossless.
3. **Promotion targets people (`profiles`), not roster rows (`students`).** Chosen because only an
   authenticated user has a role to change; roster-only records have no login.
4. **Dual presence is preserved, addressed in the UI.** Chosen so demotion never loses a student's
   reading history; the confusion cost is paid with presentation, not data movement.
5. **Admin is infrastructure-only**, enforced in both UI and server action. Chosen to keep privilege
   escalation impossible through the application surface.
6. **Teacher Management is its own module**, independent from Student Management (no generic Users
   module). Chosen to keep the roster-vs-person boundary that 12.5 established, and to give future
   teacher capabilities a natural home.

### Future teacher workflow (NOT built now)

Expected later: assigned students / classrooms / reading groups, reading-session review, student
evaluations, grading, comments, teacher statistics, teacher dashboards, teacher reports.

### Future extension points (how those attach without redesign)

- A teacher is already a first-class **`profiles.id`** identity, so future relationships key straight
  off it: e.g. `teacher_student_assignments(teacher_profile_id → profiles.id, student_id → students.id, …)`,
  `evaluations(teacher_profile_id, student_id, …)`, `teacher_profiles(profile_id PK, department, …)`.
  All are **separate extension tables**, added in their own phases.
- Resolution stays **by id** (`profiles.id` / `students.id` / `profile_id`), never email — consistent
  with the 12.5 invariant.
- Future teacher-scoped reads layer on top of the existing `requireRole`/capability-helper pattern;
  no new identity plumbing is required.

### What future phases will build on top of this

Teacher-facing features will assume: (a) `profiles.role='teacher'` is the only definition of a
teacher; (b) a teacher is a normal authenticated profile; (c) any teacher-specific data or
relationships live in extension tables keyed by `profiles.id`.

### What must NEVER be broken in future phases

- **Never** add a parallel "is a teacher" flag/table that can disagree with `profiles.role`.
- **Never** make role management write to identity columns (`profile_id`) or to `students` /
  `auth.users` / `user_settings` / `reading_sessions`.
- **Never** expose an in-app path to create, assign, or escalate to **admin**.
- **Never** delete or move a student's `students` row / reading history when changing a role.
- **Never** resolve teachers or students by email instead of by id.
- **Never** promote a roster-only record (no `profiles`) — promotion requires an authenticated user.
