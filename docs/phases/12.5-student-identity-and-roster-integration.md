# Phase 12.5 — Student Identity & Roster Integration

> ✅ **Status: Completed and accepted.** Implementation, manual testing, verification, and the
> architectural decisions below are all approved. For how the system works *today* across all phases,
> see [`docs/project/current-architecture.md`](../project/current-architecture.md) (single source of
> truth). Manual Supabase config is in
> [`docs/database/manual-supabase-configuration.md`](../database/manual-supabase-configuration.md).

## Goals

Unify the two previously disconnected student-creation flows (public registration created a login
with no roster record; admin creation created a roster record with no login) into **one identity
model**, so every active student has a complete, linked identity and the reading features actually
work end-to-end. Public-first, runs on **Supabase default auth (no SMTP, no email confirmation)**,
and usable immediately as a live recruiter-facing demo. **No schema changes.**

## Architecture

The `students.profile_id` bridge and all FK `ON DELETE` semantics already existed and are correct,
so Phase 12.5 is **entirely an application-layer effort** — no schema, policy, or privilege change
was needed. The service-role privileges it relies on (`SELECT`/`INSERT`/`UPDATE` on `public.students`)
were verified already present in the live project, so **no grant had to be run** (see the Database
changes section below and `docs/database/phase-12.5-identity-alignment.md`).

```
auth.users.id ──1:1── profiles.id ──0..1── students.profile_id ──1:N── reading_sessions.student_id
 (identity)            (role + identity)     (the bridge, SET NULL)      (CASCADE — history anchor)
auth.users.id ──1:1── user_settings.user_id (trigger-created)
```

All privileged, server-validated operations run through the **service-role admin client**
(`lib/supabase/admin.ts`) and resolve a student strictly by the permanent link (`profile_id`),
never by email.

## Identity model & invariant

The permanent identity relationship is **`auth.users.id ↔ profiles.id ↔ students.profile_id`**
(documented in `.claude/rules/architecture.md` → "Student identity invariant"). Resolve a student
by `profile_id`, never by email. Email is only authentication, communication, and *initial roster
matching before linking*; changing it never affects identity.

## Lifecycle (all states representable without a schema column)

| State | auth.users | profiles | user_settings | students | profile_id | Login |
|---|:--:|:--:|:--:|:--:|:--:|:--:|
| Roster-only (admin pre-created) | – | – | – | ✅ | NULL | No |
| Active (public self-registered) | ✅ | ✅ | ✅ | ✅ | set | Yes |
| Invited (admin activation link, not yet activated) | ✅ | ✅ | ✅ | ✅ | set | After set-password |
| Unlinked academic record (post account deletion) | – | – | – | ✅ | NULL | No (history kept) |

Status is **derived**: `roster_only` = `profile_id IS NULL`; `invited` vs `active` by
`auth.users.last_sign_in_at`.

## Registration flow (expanded, one-step) — public `signUp()`

`features/auth/components/register-form.tsx` collects Full Name, **Arabic First/Last Name**, Email,
**Grade**, Password. Public registration is **user-initiated**, so `signUpAction`
(`features/auth/actions.ts`) uses `supabase.auth.signUp()` — *not* `admin.createUser()` (that API is
reserved for admin provisioning). With "Confirm email" disabled, `signUp()` auto-confirms and
establishes the session in one step (no separate `signInWithPassword`, no `email_confirm` flag), and
it enforces the platform's signup protections.

1. Reconcile by email *first* (service-role). A roster row already **linked** → email taken (block).
   An **unlinked** roster row → leave the new account unlinked (the UNIQUE email constraint forbids a
   duplicate, and email never auto-claims) for the secure claim.
2. **`signUp()`** → creates `auth.users`, auto-confirms, establishes the session. Duplicate email →
   `emailTaken`; missing session (confirmation-required misconfig) → roll back.
3. Create `profiles` (`role='student'`) — service-role.
4. Create `user_settings` **explicitly** — via the **session client** (the signed-in user inserts its
   own row under `settings_insert_own` RLS), so every account is complete (no trigger, no deferred
   save).
5. If no roster collision, create the linked `students` row with an auto `student_number`
   (service-role).
6. Redirect to dashboard. Any insert failure rolls back (delete auth user → cascades
   profiles/user_settings; clear session).

## Claim flow (secure, no SMTP, no email verification)

`claimStudentRecordAction` + `features/students/identity/components/claim-form.tsx` (rendered on the
reading-sessions unlinked state). A signed-in student links an **unlinked** roster row using its
school-issued **`student_number`** (the claim secret) — never by email. Guarded by `profile_id IS
NULL` so concurrent claims can't both win; an already-linked student, an unknown/claimed number, or
a lost race all return a safe localized error.

## Admin provisioning (activation link, no SMTP)

`generateStudentActivationLinkAction` + `ActivationLinkDialog`. Uses Supabase
`generateLink({ type:"recovery" })` (**no email sent**) and builds a copyable link to **our own
callback** with the returned `hashed_token`
(`…/api/auth/callback?token_hash=…&type=recovery&next=/<locale>/reset-password`) the admin shares; the
student opens it to set a password (reuses the Phase 5 reset-password flow). Provisions the account
first when needed (create auth user → profile → set `profile_id`) with compensation on failure.

> **Callback supports two flows (`app/api/auth/callback/route.ts`).** Admin-generated links have **no
> PKCE code verifier** (created server-side via the Admin API), so the default `?code` exchange can't
> establish a session and the link would dead-end at Login. The callback therefore handles **both**:
> PKCE `?code` → `exchangeCodeForSession` (Phase 5 user-initiated forgot-password) **and**
> `?token_hash=&type=recovery` → `verifyOtp` (admin activation links — no verifier required). See
> `docs/project/current-architecture.md` §8/§8a. *(Fix applied after Phase 12.6; identity model
> unchanged — only the link-building tail of the action and the callback changed.)*

## Email management (admin-only, dual atomic-with-compensation)

Email is read-only on every student-facing surface (Settings has no email field). Only admin/teacher
edit it, in Student Management. `updateStudentAction` → `changeLinkedStudentEmail`: for a linked
student it updates **`auth.users.email`** (Admin API, no `email_confirm`) then **`students.email`**,
restoring the previous auth email if the second step fails — `profile_id`/ids never change, no new
identity is created. A true cross-schema SQL transaction isn't possible (auth lives behind the Admin
API), so this application-level operation with compensation is the chosen approach.

## Reconciliation backfill (one-time, admin)

`reconcileStudentLinksAction` + `ReconcileDialog` (admin only, dry-run first). Links legacy unlinked
student-profiles to existing unlinked roster rows **by email** (an admin-vouched administrative
match, distinct from the self-service claim). Idempotent; reports linked / conflicts / unmatched. It
cannot create rows for unmatched profiles (`grade` is `NOT NULL` and unknown).

## Security decisions

- **No SMTP, no email confirmation.** Public self-registration creates users **without**
  `email_confirm`, so the email is honestly *unverified*; login works because the project's "Confirm
  email" setting is OFF (a Supabase dashboard setting — see the Manual checklist).
- **Email never claims a record.** Roster claiming requires the school-issued `student_number`
  (high-entropy, app-generated) or an admin activation link — possession of a secret, not a guessable
  email. The UNIQUE `students.email` constraint forces the secure path on collision.
- Admin/teacher email change is privileged (role-gated, service-role, server-side).
- The broader tightening of the currently permissive (`using(true)`) RLS policies remains **Phase 17
  (Security Review)** scope.

## Database changes

**None required.** Phase 12.5 makes no schema/policy/trigger/function/constraint/index changes. Its
service-role operations need `SELECT`/`INSERT`/`UPDATE` on `public.students`, and those privileges
were **verified already present** in the live project (`service_role` holds all three) — so no grant
needs to be run. `auth.users` mutations use the GoTrue Admin API (no table grant). The `user_settings`
auto-create trigger is relied on, unchanged. See `docs/database/phase-12.5-identity-alignment.md` for
the verification query and details (it documents the verified state; nothing to apply).

## Testing

Build, lint (only the pre-existing TanStack `useReactTable` warning), and typecheck pass. Manual
coverage: one-step registration creates all four rows + auto number + link; collision routes to
claim (no duplicate); claim links by `student_number` only (email never claims); activation link
works without SMTP; admin email change keeps `auth.users.email == students.email` and rolls back on
partial failure; `profile_id`/ids never change on email edit; no student-facing path edits email;
reconcile dry-run/apply idempotent; account-status column reflects roster-only/invited/active; RTL +
dark mode + a11y. (See the implementation report's Manual Verification Checklist.)

## Final implementation summary

The two flows are unified behind one identity model with `profile_id` as the permanent link and
email demoted to a managed, admin-only attribute. Public self-registration is one step and fully
self-serve (no infra); school rosters integrate via a secure `student_number` claim or admin
activation links — all SMTP-free. Zero schema changes and **zero database actions** — the
service-role privileges it relies on were verified already present, so no grant had to be run.
